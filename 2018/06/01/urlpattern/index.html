关于tomcat servlet的url-pattern匹配规则

发表于 2018-06-01 |

本文引用自https://www.cnblogs.com/fangjian0423/p/servletContainer-tomcat-urlPattern.html和https://www.cnblogs.com/canger/p/6084846.html

  1. servlet容器中的匹配规则既不是简单的通配，也不是正则表达式，而是特定的规则。所以不要用通配符或者正则表达式的匹配规则来看待servlet的url-pattern。
  2. servlet2.5开始，一个servlet可以使用多个url-pattern规则。
  3. 当servlet容器收到浏览器发起的一个url请求以后，会用url减去当前应用的上下文路径,用剩余的字符串作为servlet映射，假如url是http://localhost:8080/appDemo/index.html，其应用上下文是appDemo，容器会将http://localhost:8080/appDemo去掉，用剩下的/index.html部分拿来做servlet的映射匹配。
  4. url-pattern映射匹配过程是有优先顺序的，而且当有一个servlet匹配成功以后，就不会去理会剩下的servlet了。

    四种匹配规则

    1.路径匹配

    以”/*”结尾的url-pattern

    2.扩展名匹配

    以”*.”开始的url-pattern

    3.缺省匹配

    默认是”/“

    4.精确匹配

    除了以上三种之外的其他的url-pattern

分析

tomcat启动过程中，WebXml会负责扫描web.xml文件，得到servlet的映射数据 ServletMappings,然后会调用Context(实现类为StandardContext)的addServletMapping方法。 这个方法会调用Mapper的addWrapper方法

各种对应的处理完成之后，会存入context的各种wrapper中。这里的context是ContextVersion，这是一个定义在Mapper内部的静态类。它有4种wrapper。 defaultWrapper，exactWrapper， wildcardWrappers，extensionWrappers。这里的Wrapper概念：Wrapper 代表一个 Servlet，它负责管理一个 Servlet，包括的 Servlet 的装载、初始化、执行以及资源回收。
断点调试ContextVersion，可以看到

  1. url-pattern为/helloworld的exacttest被丢到了exactWrappers中
  2. url-pattern为/的default被丢到了defaultWrapper中
  3. url-pattern为/login/*的wildcardtest被丢到了wildcardWrappers中
  4. url-pattern为*.do的extensiontest被丢到了extensionWrappers中，里面还有resourceOnly属性为true的jsp wrapper是用来处理jsp资源的。

    当用户请求进来的时候会调用Mapper的internalMapWrapper方法

用户请求匹配MappedWrapper的优先级如下

  1. 规则1：精确匹配，匹配contextVersion的exactWrappers
  2. 规则2：路径匹配，匹配contextVersion的wildcardWrappers
  3. 规则3：扩展名匹配，匹配contextVersion的extensionWrappers
  4. 规则4：使用contextVersion的welcomeResources属性匹配欢迎界面例如index,welcome等资源。
  5. 规则5：默认匹配，使用contextVersion的defaultWrapper。

    注意

  6. “/”属于路径匹配，并且可以匹配所有request，由于路径匹配的优先级仅次于精确匹配，所以“/”会覆盖所有的扩展名匹配，很多404错误均由此引起，所以这是一种特别恶劣的匹配模式，一般只用于filter的url-pattern
  7. “/”是servlet中特殊的匹配模式，切该模式有且仅有一个实例，优先级最低，不会覆盖其他任何url-pattern，只是会替换servlet容器的内建default servlet ，该模式同样会匹配所有request。
